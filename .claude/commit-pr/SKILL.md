# commit-pr — 커밋 / PR 작성

커밋·PR 메시지를 직접 다듬을 필요 없게, **아래 규칙 그대로** 작성한다.
장황한 설명·이모지·과한 포맷은 거슬려한다. 짧고 사실만.

## 공통 원칙

- **이모지 금지.** 🤖 / ✅ / ⚠️ / 🚨 등 절대 넣지 않는다.
- **없어도 되는 건 넣지 않는다.** 자잘한 디테일·과정 설명·"~하였습니다" 나열 금지.
- 실제로 한 일과 이유만. 추측·홍보성 문구 금지.

## 커밋

한 줄 subject가 기본. 무엇보다는 '왜'를 잘 파악할 수 있게. 본문은 필요 시 작성.

- 형식: `prefix(scope): 요약` — scope는 있으면 붙이고 없으면 생략
  - prefix: `feat` `fix` `refactor` `chore` `docs` `test`
  - scope 예: `feat(sdk):`, `feat(labnosh):`, 플랫폼/고객사/모듈명
- subject는 **무엇을/왜** 한 줄. 명사형·간결하게. (예: `fix(sdk): 오픈 방식 추적을 위해 eventType 업데이트`)
- 본문이 필요하면 빈 줄 뒤 불릿 몇 개. 개조식. 산문 문단 지양.
- **커밋 메시지 끝에 트레일러 유지** (하네스 필수, 이모지 아님):
  ```
  Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>
  ```

## PR

밀도는 **간결 (배경 + 변경만).**

```
## 배경
- <왜 이 작업을 하는지 1>
- <왜 이 작업을 하는지 2>

## 변경
- <바뀐 것 1>
- <바뀐 것 2>
```

- **배경·변경 모두 불릿 개조식으로 작성한다.** 산문 문단 지양. 배경이 자명하면 생략 가능.
- **검증/테스트/관련 레포/표/동작흐름 섹션은 기본적으로 넣지 않는다.** 정말 중요한 주의사항이 있을 때만 한 줄 덧붙인다.
- 제목은 커밋 subject와 같은 형식 (`prefix(scope): 요약`).
- **🤖 Generated with Claude Code 푸터는 넣지 않는다.**
- `gh pr create`로 올린다.

## 이 레포 git 규칙 (CLAUDE.md)

- **브랜치 작업 권장.** 작업은 `feat/*`·`fix/*`·`refactor/*` 브랜치에서 하고 PR로 merge. 현재 main이면 먼저 브랜치를 판다.
